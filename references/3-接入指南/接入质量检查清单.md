# 接入质量检查清单

本文档用于评估汇聚支付接入质量。按以下清单逐项检查，标记通过/不通过/不适用。

## 一、签名验签检查

### MD5签名（如使用）

- [ ] 参与签名的参数排除了 `hmac` 字段
- [ ] 参数按 key 的**字母升序**排序（不是自然排序）
- [ ] 只拼接 value，不拼接 key 名
- [ ] 拼接后在末尾追加了 **merchantKey**
- [ ] MD5 结果转为**大写**
- [ ] 编码统一使用 **UTF-8**

### RSA签名（如使用）

- [ ] 私钥格式为 **PKCS8**（不是PKCS1）
- [ ] 公钥上传到了汇聚商户后台
- [ ] 验签使用的是**商户公钥**（不是平台公钥）
- [ ] Base64解码时做了**空格→加号**替换
- [ ] 公钥长度 > 1024 时使用 **X.509** 格式解析

### 回调解签

- [ ] 回调接口同样进行了验签（**必须！**）
- [ ] 验签通过后才进行业务处理
- [ ] 验签失败返回 `fail`，不抛异常

## 二、参数完整性检查

### 下单接口

- [ ] p1_MerchantNo（商户号）— 正确填写
- [ ] p2_OrderNo（订单号）— 全局唯一，仅英文数字下划线
- [ ] p3_Amount（金额）— 两位小数，大于0
- [ ] p5_ProductName（商品名称）— 非空
- [ ] q1_FrpCode（交易类型）— 符合规范
- [ ] p9_NotifyUrl（回调地址）— 公网可达
- [ ] qa_TradeMerchantNo（报备商户号）— 如已报备则填写

### 渠道特定参数（按实际使用的FrpCode检查）

- [ ] **GZH/XCX**: q5_OpenId + q7_AppId 都已传
- [ ] **CARD（被扫）**: q6_AuthCode + q8_TerminalNo + ql_TerminalIp 都已传
- [ ] **APP/APP3/GZH/XCX**: q7_AppId 已传
- [ ] **FWC**: qb_buyerId 已传

## 三、业务逻辑检查

- [ ] **幂等处理**：同一订单号不会重复创建订单
- [ ] **金额校验**：前端传入的金额与后端数据库一致（防篡改）
- [ ] **状态检查**：退款前检查原订单确实处于已支付状态
- [ ] **超时处理**：HTTP请求设置了合理的超时（建议15-30s）
- [ ] **异常捕获**：网络异常、超时有明确的异常处理和重试机制
- [ ] **日志记录**：关键操作（下单/支付/退款/回调）都有完整的请求/响应日志

## 四、回调处理检查

- [ ] **快速响应**：回调接口在 **5秒内** 返回
- [ ] **返回格式**：成功返回纯文本 `success`（无HTML、无BOM）
- [ ] **幂等防重**：收到重复回调不会重复发货/重复记账
- [ ] **IP白名单**：（如有配置）放行了汇聚的服务器IP
- [ ] **HTTPS**：生产环境使用 HTTPS（推荐）
- [ ] **通知确认**：收到回调后建议主动调用一次查询接口确认

## 五、安全性检查

- [ ] **密钥存储**：商户密钥/私钥不在代码中硬编码，存放在配置文件或密钥管理服务
- [ ] **敏感信息脱敏**：日志中不打印完整密钥、完整付款码
- [ ] **SQL注入防护**：订单号等外部输入做了参数化处理
- [ ] **金额精度**：使用 BigDecimal（Java）/ Decimal（Python/PHP）处理金额，不用 float/double

## 六、测试覆盖

- [ ] **正向测试**：各支付渠道的下单→支付→回调全流程跑通
- [ ] **签名错误测试**：故意篡改参数验证验签能正确拦截
- [ ] **金额边界测试**：0.01元、大额、两位小数边界
- [ ] **退款测试**：部分退款、全额退款、超额退款拒绝
- [ ] **关单测试**：未支付订单关单、已支付订单关单（应失败）
- [ ] **重复请求测试**：相同订单号重复提交

## 评分标准

| 等级 | 通过率 | 评级 |
|------|--------|------|
| 优秀 | ≥ 95% | 🟢 A级 — 可以放心上线 |
| 良好 | ≥ 85% | 🟡 B级 — 建议修复后再上线 |
| 及格 | ≥ 70% | 🟠 C级 — 存在明显隐患，强烈建议修复 |
| 不及格 | < 70% | 🔴 D级 — 存在严重问题，禁止上线 |

---

> **人设提示**：作为金融系统质检专家，对每项检查给出明确结论。发现问题时不仅指出"什么错"，还要解释"为什么错"和"怎么改"。对于涉及资金安全的项（签名、金额、幂等），标注为 ⚠️ 致命级别问题。
