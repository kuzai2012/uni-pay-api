# 排障手册

## TOP 20 响应码速查表

以下是汇聚支付最常见的 20 个错误响应码，按出现频率排列：

| 排名 | 响应码 | 含义 | 快速排查方向 |
|------|--------|------|-------------|
| 1 | 10080000 | **签名验证失败** | ①参数是否完整参与签名 ②key是否正确 ③大小写 ④编码UTF-8 |
| 2 | 10080001 | **参数为空** | 检查rb_CodeMsg中的具体字段名 |
| 3 | 10080002 | **商户号不存在** | 确认p1_MerchantNo是否正确 |
| 4 | 10080003 | **商户号未激活** | 联系汇聚客服激活商户 |
| 5 | 10080005 | **订单号重复** | 更换新的订单号（幂等问题） |
| 6 | 10080006 | **订单金额有误** | 检查格式（两位小数）、范围（>0） |
| 7 | 10080008 | **商品名称为空** | p5_ProductName不能为空 |
| 8 | 10080009 | **交易类型有误** | q1_FrpCode拼写错误或不在支持的列表中 |
| 9 | 10080012 | **商户未开通此支付方式** | 到汇聚后台申请开通对应的支付产品 |
| 10 | 10080016 | **交易已存在** | 相同订单号重复提交，先查询订单状态 |
| 11 | 10080017 | **订单号格式有误** | 仅允许英文、数字、下划线，最长50字符 |
| 12 | 10080020 | **查询不到交易记录** | 订单号不存在或尚未创建 |
| 13 | 10080021 | **交易已关闭** | 已关闭的订单无法操作 |
| 14 | 10080028 | **Appid不能为空** | GZH/XCX/APP/H5+必须传q7_AppId |
| 15 | 10080029 | **Appid未报备** | AppId需要在汇聚后台完成报备 |
| 16 | 10080038 | **限流** | 过去一秒内请求量太大，降低频率后重试 |
| 17 | 10083001 | **渠道下单失败** | 查看rb_CodeMsg中的渠道具体错误码 |
| 18 | 110 | **系统正忙** | 汇聚系统繁忙，稍后重试 |

| 20 | 20090007 | **退款订单号重复** | 换一个唯一的退款订单号 |

## 排障定位流程

```
遇到错误
  │
  ├─ 有响应码(ra_Code/rb_Code)?
  │   │
  │   ├─ 是 → 查TOP 20速查表
  │   │       ├─ 匹配 → 按快速排查方向处理
  │   │       └─ 未匹配 → 加载[📄响应码完整列表](./响应码完整列表.md)
  │   │
  │   └─ 否 → 进入路径B（异常诊断）
  │
  ▼
路径B: 异常诊断
  │
  ├─ 网络错误?
  │   └─ 检查base_url、DNS、防火墙、SSL证书
  │
  ├─ 超时?
  │   └─ 检查网络延迟、服务器负载、请求体大小
  │
  ├─ 解析失败?
  │   └─ 检查响应是否为有效JSON/form格式
  │
  └─ 以上都正常但仍异常?
      └─ 加载[📄基础支付常见问题](./基础支付支付常见问题.md)
```

## 场景式排障

### 场景A：签名失败 (10080000)

这是**最常见**的错误，按以下步骤逐一排查：

1. **打印参与签名的参数列表**
   ```python
   # 调试：输出参与签名的所有参数
   print("参与签名的参数:", {k:v for k,v in params.items() if k.lower()!='hmac'})
   ```

2. **检查排序结果**
   ```
   正确排序示例: p0_Version, p1_MerchantNo, p2_OrderNo, p3_Amount ...
   ```

3. **手动复现拼接**
   ```
   signStr = "2.6" + "你的商户号" + "订单号" + "0.01" + ... + "商品名称"
   hmac = MD5(signStr + 你的密钥).toUpperCase()
   ```

4. **常见坑点**
   - ❌ 包含了 `hmac` 字段参与签名
   - ❌ 按 value 排序而非 key 排序
   - ❌ 拼接了 key=value 而不是只有 value
   - ❌ 忘记追加 merchantKey
   - ❌ 大小写不一致（应为UPPER）
   - ❌ 编码问题（GBK vs UTF-8）

### 场景B：下单成功但收不到回调 / 回调地址校验失败

这是**本地开发最常遇到**的问题：

#### 现象

下单接口直接返回错误：
```json
{
  "ra_Code": "101",
  "rb_CodeMsg": "传递参数有误：[p9_NotifyUrl=http://localhost:8080/notify/pay;不是正确的URL]"
}
```

#### 原因

汇聚服务端会在下单时**主动校验 p9_NotifyUrl 格式**，拒绝无法从公网访问的地址。

#### 被拒绝的地址类型

| 地址类型 | 示例 | 为什么被拒 |
|---------|------|-----------|
| `localhost` | `http://localhost:8080/notify` | `localhost` 指向的是汇聚服务器的本机(127.0.0.1)，不是你的电脑 |
| `127.0.0.x` | `http://127.0.0.1:8080/notify` | 同上，回环地址只有本机能访问 |
| 内网IP | `http://192.168.1.100:8080/notify` | 汇聚服务器在你的内网外，无法访问 |
| 空值/非URL | `` 或 `abc` | 不是合法的 HTTP/HTTPS URL |

#### 解决方案

**方案1：使用内网穿透工具（推荐用于本地开发）**

```bash
# ngrok（最简单，免费）
ngrok http 8080
# 输出: https://xxxx.ngrok-free.app -> http://localhost:8080
# 然后把 https://xxxx.ngrok-free.app/notify 作为回调地址

# cpolar（国内速度更快）
cpolar http 8080

# frp（自建，稳定可靠）
# 需要一台有公网IP的服务器运行 frps
```

**方案2：使用测试服务器/生产环境域名**

```
测试环境: http://10.x.x.x:8080/api/pay/notify    # 如果汇聚能访问该内网
生产环境: https://pay.yourdomain.com/api/pay/notify  # 推荐 HTTPS
```

> **注意**: 如果你的测试环境在汇聚可访问的内网(如同一机房)，内网IP可能可用。但 `localhost` 和 `127.0.0.1` **永远不可用**。

#### SDK 已内置前置校验

所有语言的 SDK 都已内置 `validateNotifyUrl()` 方法，在发送请求到汇聚之前就会拦截非法地址并给出明确提示：

```
[Java]   IllegalArgumentException: 回调地址不能使用 localhost！...
[Python] ValueError: 回调地址不能使用 localhost！...
[Go]     error: 回调地址不能使用 localhost！...
[PHP]    InvalidArgumentException: 回调地址不能使用 localhost！...
```

这样开发者可以在本地开发阶段就发现问题，无需等到服务端报错后再排查。

### 场景C：渠道下单失败 (10083001)

这个错误表示汇聚本身没问题，但**上游渠道**返回了错误：

1. 查看 `rb_CodeMsg` 中嵌套的渠道错误信息
2. 常见渠道级错误：
   - **微信**: appid未绑定 / openid无效 / 余额不足
   - **支付宝**: 协议已失效 / 买家账户受限
   - **银联**: 卡号无效 / 限额超限

### 场景D：退款失败

1. 先确认**原订单状态**：必须是 SUCCESS
2. 确认**剩余可退金额**：用 queryRefundInfo 接口查询
3. 检查**退款订单号**是否唯一
4. 检查**签名**：退款接口也需要正确签名

## 排障辅助脚本

本技能提供两个排障脚本：

| 脚本 | 用途 | 使用场景 |
|------|------|---------|
| `scripts/query_order.py` | 查询订单详细状态 | 排查支付结果、回调丢失 |
| `scripts/query_refund.py` | 查询退款详细状态 | 排查退款进度、退款失败原因 |

> 使用脚本前需按**分步确认协议**征得用户同意，并引导用户提供必要的认证信息（不记录密钥）。
